# Audit fable-collector — juillet 2026

Audit réalisé le 2026-07-05 sur `rbpower-hub/fable-collector` (dernier commit
audité : `b4343ef`, 2025-10-07). Toutes les corrections listées sont intégrées
dans la refonte v2.x livrée avec ce rapport, base v2.0.0 complétée ensuite
par la version v2.1.0.

## 1. Constat principal : production arrêtée depuis 7 mois

Le site Pages servait des prévisions générées le **2025-12-06 18:13** (vérifié
sur `status.json` en production le 2026-07-05).

Cause racine : GitHub **désactive automatiquement les workflows `schedule`
après 60 jours sans activité du dépôt**. Dernier commit le 07/10/2025 →
désactivation début décembre 2025, soit exactement la date du dernier
déploiement. Personne n'a été alerté car (§2, §3) le monitoring ne pouvait
structurellement pas détecter la panne.

**Corrections v2 :**
- Job `keepalive` hebdomadaire dans `collect.yml` : commit vide tous les
  dimanches 03:23 UTC → le dépôt n'est plus jamais « inactif » aux yeux de
  GitHub (`chore: keepalive [skip ci]`).
- Healthcheck externe réel (§3) qui ouvre une **issue GitHub** si la prod
  devient obsolète, quelle qu'en soit la cause.

## 2. status.json mentait (« all_fresh: true » pendant la panne)

La fraîcheur était calculée **au moment du build** : les fichiers venant
d'être écrits, `all_fresh` était vrai par construction, puis figé tel quel
sur Pages. Un statut statique ne peut pas se périmer lui-même.

**Correction v2 :** `status.json` publie `generated_at` + `stale_after`
(cadence 60 min + 35 min de tolérance) ; `status.html` évalue l'âge
**côté client** (JS, horloge du visiteur) et affiche ❌ OBSOLÈTE dès que la
limite est dépassée. La surveillance serveur est assurée par le healthcheck.

## 3. healthcheck.yml n'a jamais tourné

Le fichier était à la **racine** du dépôt au lieu de `.github/workflows/` —
GitHub ne le voyait pas. Le badge README pointait vers un workflow inexistant.
De plus sa logique lisait `index.json` en attendant le schéma de
`catalog.json` (clé `files` vs `spots`) : même déplacé, il aurait échoué à
chaque run.

**Correction v2 :** `.github/workflows/healthcheck.yml` (cron /6 h) exécute
`fable/healthcheck.py` : fraîcheur de `status.json` (< 95 min), présence et
complétude de chaque spot (≥ 24 points horaires), cohérence de
`windows.json`, liste des spots attendus lue depuis `sites.normalized.json`
en production (auto-descriptif). En cas d'échec : issue GitHub ouverte/
commentée (label `healthcheck`) + badge rouge. Plus besoin de secrets SMTP.

## 4. windows.json polluait les destinations

Le reader globbait `public/*.json` en n'excluant que `index.json` et
`windows.json` → `catalog.json` et `rules.normalized.json` apparaissaient
comme **destinations de sortie en mer** dans le JSON publié (vérifié en prod).

**Correction v2 :** reconnaissance des spots **par contenu** (`meta` +
`hourly.time` non vide) et non par liste noire de noms. Testé
(`test_run_reader_ignores_non_spot_files`) y compris avec un JSON inconnu.

## 5. Cron inefficace et fragile

`cron: "0 * * * *"` + gate « heure locale ∈ {00,06,12,18} » :
- 20 runs/jour démarraient un runner pour s'auto-annuler ;
- le cron GitHub ayant souvent 10-40 min de retard, un déclenchement à
  06:47 passait (heure=06) mais un à 07:05 était **sauté** → collectes
  manquées aléatoirement ;
- le README promettait un collecteur « horaire » — la réalité était 4×/jour.

**Correction v2 :** cron horaire direct `7 * * * *` sans gate (la Tunisie
n'ayant pas de DST, aucune conversion fragile n'est nécessaire). Décision
produit : fraîcheur horaire réelle (~720 appels API/jour pour 5 spots,
largement sous la limite gratuite Open-Meteo de 10 000/jour).

## 6. Échec marine = site entier perdu

Dans `main.py`, un échec de l'API Marine (ou un dépassement du budget temps)
levait une exception qui **sautait tout le site** : les données de vent déjà
collectées étaient jetées.

**Correction v2 :** `fetch_marine` ne lève jamais — il renvoie un payload
dégradé `{"hourly": {}, "_error": ...}` ; le spot est publié **sans vagues**
avec `meta.debug.marine_error`, et le détecteur classe automatiquement ces
heures non-éligibles (`vagues_inconnues`). Aucune donnée fabriquée. Testé.

## 7. Fenêtres 7-8 h avec phases erronées possibles

Le modèle de phases Transit–Mouillage–Transit est défini pour 4-6 h, mais
`--max-hours` valait **8** par défaut et le workflow ne passait pas l'option.
Une fenêtre de 7-8 h voyait ses heures 7-8 retomber en « transit » par le
fallback d'indice — incohérent avec le design.

**Correction v2 :** bornes lues depuis `rules.yaml` (`window_hours`, défaut
4-6) et **plafond dur à 6 h** dans `run_reader` (avec warning). Testé.

## 8. Seuils dupliqués en 4 endroits (dérive avérée)

`rules.yaml`, défauts de `main.py`, défauts de `reader.py` et constantes
hardcodées de `reasons-debug.js` — avec des valeurs déjà divergentes
(ex. Tp@Hs<0.4 : 3.2 s dans rules.yaml et reader.py, 4.0 s dans main.py).
Les secteurs onshore étaient dupliqués reader.py + JS. Ligne 174 de
reader.py : `float(_dget(...) and 0.35) or 0.35` — expression morte évaluant
toujours 0.35.

**Correction v2 :** `fable/config.py` est l'unique source de défauts ;
collector et reader consomment le même objet règles ; les seuils
mers courtes/raides (hardcodés v1) sont désormais dans les règles
(`combined.*`) et exportés dans `rules.normalized.json` ; les secteurs
onshore vivent dans `sites.yaml` (v2), sont republiés dans chaque
`meta.onshore_sectors` et dans `sites.normalized.json` (consommables par le
dashboard). L'expression morte a disparu avec la refonte.

## 9. Multi-port

Politique d'exclusion (`korbous`, `kelibia`) et port d'attache étaient
hardcodés dans le code Python.

**Correction v2 :** `sites.yaml` v2 : `home`, `exclude`, `defaults`,
`onshore_sectors` par site. Déployer pour un autre port = éditer un fichier.
Rétro-compatibilité v1 testée (liste simple, exclusions et secteurs legacy).

## 11. Compatibilité consommateurs (FABLE AI)

Le schéma de sortie v2 est **rétro-compatible** : toutes les clés v1
(`meta`, `ecmwf`, `marine`, `daily`, `daily_units`, `hourly`, `models`,
`forecast_primary`, `status`, toutes les séries horaires) sont préservées —
vérifié par diff structurel contre le payload de production d'octobre 2025.
Ajouts uniquement additifs : `meta.collector_version`, `meta.onshore_sectors`,
`sites.normalized.json`, `status.stale_after`, `windows.window_hours`,
`windows.rules_digest`.

Seule rupture assumée : `index.json` n'a plus le double schéma
`index.json`/`index.spots.json` (le schéma réel observé en prod — `spots` —
est conservé et enrichi de `home` et `collector_version`).
